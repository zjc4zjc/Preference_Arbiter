# 🚀 Agent-Centric Research Protocol（科研流程协议·当前版本）

## 0. 核心哲学（Philosophy）

- **Agent First**：除非是方向选择、价值判断、科学直觉，其他能自动化的事情都交给 Agent。  
- **Everything is Markdown**：`Discussion.md`、 `Idea.md`、`Log.md`、 `Method.md`、`Record\`  是整个研究的“外部大脑”，所有重要信息都必须最终落在这几个文件里。  
- **Vibe Alignment**：人类负责定义 Vibe（想解决什么问题、往哪儿走），Agent 负责把 Vibe 变成具体代码、实验和文档。

---

## 1. 项目锚点文件（Project Anchors）

本仓库中，Agent 必须认识并优先使用的核心文件有：

- `Discussion.md`  
  - 定义：围绕“当前一个具体问题”的轻量讨论模板。  
  - 用法：  
    - 顶部写清“当前问题（一句话）”；  
    - 每条发言一行头部：`【角色/姓名】【YYYY-MM-DD HH:MM】`，下一行开始写内容。  
  - 目标：让“我们现在到底在纠结什么”一眼可见。

- `Idea.md`  
  - 定义：整个项目的 **研究蓝图 / 立项书模板**。  
  - 内容：背景、SOTA、研究目标、整体实验设计与指标。  
  - 要求：**在做任何实质性修改或实验前，先完整阅读一次。**

- `Log.md`  
  - 定义：**按周组织的实验周志模板**。(Log仅为参考，实际命名应服从规范)  
  - 命名：`YYYY.MM.DD~YYYY.MM.DD.md`（例如 `2026.03.09~2026.03.16.md`），一周一个文件。  
  - 内容：本周 Vibe、每个实验的记录（EXP-...）、本周总结 & 下周 Vibe。  
  - 要求：每当跑完一组有意义的实验，Agent 必须在对应周文件中追加一条记录。

- `Method.md`  
  - 定义：**数学视角的方法说明书模板**。  
  - 内容：问题形式化、损失函数、定理/推论、为什么奏效、如何和实验对上。  
  - 要求：任何影响方法本身的改动（新损失、新正则、新假设）都要在这里留下痕迹。

- （预留）`TODO_AGENT.md`  
  - 定义：Agent 的任务看板（可以后续按需添加）。  
  - 建议格式：`[ ] Pending` / `[/] In Progress` / `[x] Done`。

- `Record/`  
  - 定义：存放实际文件的文件夹 
  - 注意：`Template/` 文件夹里仅为模板，具体内容的文件应存放至 `Record/` 文件夹里

## 1.1 代码与参考资料目录（Code & References）

- `Code/`  
  - 存放本项目的主要代码实现（模型、训练脚本、评估脚本等）。

- `Baseline/`  
  - 存放 Baseline 方法的代码实现。  
  - 可以不存在；若该目录为空或缺失，Agent 不需要强行创建 Baseline。

- `Ref/`  
  - 存放与本项目相关的资料：论文 PDF、阅读笔记、截图等。  
  - **默认规则：除非用户明确要求或某个任务显式指定需要查阅某篇资料，否则 Agent 不应主动读取 `ref/` 中的内容，以避免无谓占用上下文。**

## 2. Agent 工作循环（P-E-R Loop）

在这个仓库里，每开启 / 推进一次任务，都应遵循 **P-E-R** 流程：

### 2.1 Plan（@Plan：对齐 Vibe）

1. 阅读 / 更新 `Idea.md`，确认：  
   - 当前大课题是什么？  
   - 这一阶段想解决的是哪一块（例如鲁棒性、收敛性、可解释性）？
2. 在 `Discussion.md` 里写清 **当前问题（一句话）**，并补上一条自己的发言：  
   - 说明**现在卡在哪里**、**打算先从哪一步试**。
3. （可选）在 `TODO_AGENT.md` 中加上对应的 todo 项（如果你已经创建该文件）。

### 2.2 Execute（@Execute：实现与实验）

1. 代码 / 配置层面的动作：  
   - 根据 `Idea.md` 和 `Method.md` 的设定，修改代码、脚本、配置等。  
   - **任何影响方法定义或假设的改变，优先同步到 `Method.md`。**
1. 实验层面的动作：  
   - 运行一次或多次实验；  
   - 每完成一组有意义的实验：  
     - 找到对应周的 `Record/YYYY.MM.DD~YYYY.MM.DD.md` 文件；  
     - 追加一个新的 `EXP-...` 区块，记录：  
       - 源意图（Original Vibe）  
       - 核心修改 / 配置  
       - 关键指标  
       - 简短结论 + 下一步建议。

### 2.3 Reflect（@Reflect：反思与更新）

1. 回到 `Discussion.md`：  
   - 以 Agent 的身份补一条发言，简要汇报刚刚实验的发现和链接到对应周志文件。  
1. 回到 `Method.md`（如有必要）：  
   - 如果实验结果支持或否定某个假设，更新“为什么奏效 / 局限性 / Agent Notes”等相关部分。 
1. 回到 `Idea.md`（阶段性）：  
   - 当结论会影响大方向（例如：某条研究目标已完成 / 被否决），同步更新对应章节。

## 3. 日志与溯源约定（Traceability）

为了方便未来的 Agent / 人类快速复盘，本项目要求：

- **实验粒度**：  
  - 以“回答一个具体问题 / 验证一个具体假设”为单位划分实验；  
  - 每个实验对应 `Record/YYYY.MM.DD~YYYY.MM.DD.md` 中的一个 `EXP-...` 区块。

- **范例结构（参考 `Log.md`）：**
  - 实验 ID：`EXP-001`  
  - 源意图（Original Vibe）  
  - Agent 执行动作（改了什么）  
  - 关联文件（`Method.md` / 代码路径 / `Discussion.md` 发言等）  
  - 关键指标与简明结论  
  - 下一步建议（Next Step）

- **讨论与实验的连接：**
  - `Discussion.md` 里的讨论，尽量引用对应的 `EXP-...`；  
  - 周志文件中的实验，如直接源于某条讨论，也建议在“关联文件”里记录对应发言的时间或内容摘要。

## 4. 推荐指令与用法（Shortcuts）

以下是建议给 Agent 使用的”动作模式”（不一定是真正的命令）：

- **`/init-research`**
  - 行为：这是 Agent **每次对话开始时默认执行的第一步**（见 Section 6），无需用户手动触发。
  - 具体动作：读取最近一周的 `Record/*.md`，自行判断当前科研阶段，然后用**用户能看懂的语言**输出状态摘要，并主动询问用户这次想做什么。

- **`/new-question`**
  - 行为：在 `Discussion.md` 中更新”当前问题（一句话）”，并写入一条新的起始发言，说明本轮要解决什么。

- **`/run-exp`**
  - 行为：根据当前问题和方法设定，执行一次实验，并在当前周的 `Record/YYYY.MM.DD~YYYY.MM.DD.md` 中新增一个 `EXP-...` 记录。

- **`/sync-method`**
  - 行为：当发现实验结果对理论/方法有影响时，检查 `method.md`，补充或修正相应的公式、定理、假设与解释，并在 `Agent Notes` 中记录一次时间戳。

## 6. Agent 入场行为规范（Session Start Protocol）

**这是强制行为，每次对话开始时必须执行，不需要用户触发。**

### 6.1 第一步：自主读取项目状态

对话开始后，Agent 应立即读取 `Record/` 里的以下文件（按顺序）：

1. `Idea.md` — 判断研究课题是否已定义
2. `Method.md` — 判断方法是否已形式化
3. `Discussion.md` — 了解当前正在纠结的问题
4. `YYYY.MM.DD~YYYY.MM.DD.md` 周志文件 — 了解最近的实验进展

### 6.2 第二步：自主判断当前科研阶段

根据文件内容，Agent 自行判断当前所处阶段：

| 阶段               | 判断依据                                   | 典型状态          |
| ---------------- | -------------------------------------- | ------------- |
| **阶段 0：尚未立题**    | `Idea.md` 仍是空模板，无实质内容                  | 刚开始，什么都没有     |
| **阶段 1：有想法，没方法** | `Idea.md` 有内容，`Method.md` 仍是模板         | 知道做什么，不知道怎么做  |
| **阶段 2：有方法，没实验** | `Idea.md` 和 `Method.md` 均有内容，周志文件无实验记录 | 方法设计完，还没跑过实验  |
| **阶段 3：实验进行中**   | 周志文件有实验记录，`Discussion.md` 有当前问题        | 正在推进，有具体问题待解决 |
| **阶段 4：收尾/写作**   | 多轮实验完成，主要结论已有，开始整理                     | 准备写论文或总结      |

### 6.3 第三步：主动告知用户并询问意图

Agent 必须主动开口，**用简单直白的语言**（避免专业术语堆砌）告诉用户：

1. **我看到你现在在哪一步**（一两句话说清楚）
2. **下一步通常该做什么**（给出方向性建议）
3. **你这次想做什么**（让用户选择或自由描述）

示例输出格式（根据实际情况调整，不要照抄）：

> 我看了一下项目文件。目前你已经定义了研究问题（idea.md 有内容），方法部分也有了基本框架（method.md），但还没有跑过任何实验。
>
> 通常这个阶段的下一步是：设计第一个小实验来验证方法的核心假设（Pilot Study）。
>
> 你这次想做什么？可以告诉我，比如"想开始跑第一个实验"、"想先把方法再想清楚一点"，或者随便聊聊你的想法也行。

### 6.4 第四步：根据用户回答，引导逐步推进

用户回答后，Agent 按照 P-E-R 循环执行对应动作。

**如果用户不知道该做什么**（例如回复"不知道"、"你帮我决定"、"我是新手"），Agent 应：

1. 根据当前阶段，给出**最小可行的下一步**（一件具体的事，不是一个列表）
2. 主动帮用户完成这一步，完成后再问"接下来想继续哪个方向"
3. 不要一次性抛出所有选项让用户选择——一次只推进一步

**如果项目文件全部为空模板（阶段 0）**，Agent 应通过对话逐步收集信息，而不是让用户自己填模板：

- 第一个问题：**"你大概想研究什么问题？用自己的话说就行，不需要正式。"**
- 收到回答后，Agent 帮用户提炼并写入 `idea.md` 对应段落，然后问第二个问题
- 依次收集：背景动机 → 创新点 → 对比对象 → 数据集/场景 → 评价指标
- 每收集一块就立即更新文件，让用户能实时看到进展

## 5. 边界与防御（Safety & Quality）

- **幻觉检查（尤其是数学 & SOTA）**：  
  - 若涉及定理、证明、复杂推导或与文献对比，Agent 应：  
    - 要么给出可被复查的推导步骤；  
    - 要么附上明确的外部文献引用（标题/作者/年份/大致结论）。  

- **一致性检查：**  
  - `Idea.md` 中的研究目标、`Method.md` 中的方法定义、代码实现、周志文件中的实验说明，四者之间若出现明显不一致，应优先提醒并协助对齐。  

- **环境记录（可选）：**  
  - 若涉及容易漂移的环境（不同机器、依赖版本等），建议在周志文件对应周开头简要记录环境信息，或在某个 `EXP-...` 条目中给出链接（如 `env.yaml` / `requirements.txt`）。

> 剩下的——方法细化、实验设计与记录、讨论同步和文档更新——我会尽量全自动帮你接上。
